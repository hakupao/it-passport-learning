## strategy-02-04 — 知的財産権（strategy）
- 目標: 知的財産権にはどのような種類があり，何が法律で守られ，どのような行為が違法に当たるのかの基本を理解する。
- 節点頻度: primary 69題 / secondary 17題 → **頻出**
- Rule D: **PASS**（1ラウンド: r1=PASS）
- 3 ユニット / 18 用語

### strategy-02-04-u01 著作権法と産業財産権の各法　[8語 / topic内題数計109] （前提なし）
知的財産権の二大区分である著作権法と産業財産権(特許・実用新案・意匠・商標)の各法が、それぞれ何を保護するのかを体系的に理解する。
<sub>分割根拠: objective の核心『どのような種類があり何が守られるか』に直接対応する基礎ユニット。無方式主義で自動発生する著作権法と、登録(方式主義)を要する産業財産権4法を対比して学ぶことが、以降の契約・提供形態を理解する前提となる。商標の下位概念であるトレードマーク/サービスマークは商標法の直後に置き、概念のまとまりを保つ。高頻度 term(著作権法36・特許21・意匠19・商標15)が集中し、体系理解の核としても整合。前提概念を扱うため最初に配置(prerequisites なし)。8 term は『IP 諸法＋マーク下位概念』という一体の体系であり、概念的根拠により上限8を採用。</sub>
用語順: 著作権法(36) → 特許法(21) → ビジネスモデル特許(6) → 実用新案法(9) → 意匠法(19) → 商標法(15) → トレードマーク(2) → サービスマーク(1)

### strategy-02-04-u02 不正競争の防止とソフトウェア使用許諾契約　[5語 / topic内題数計21] （前提: strategy-02-04-u01）
登録に依らない保護を担う不正競争防止法と、ソフトウェアの使用権を契約で付与・管理する各種ライセンス契約(使用許諾・ボリューム・サイト・CAL)を理解する。
<sub>分割根拠: u01 で『権利の体系』を学んだ上で、その権利を実務でどう守り・どう使わせるかに進む。不正競争防止法は登録を要さず営業秘密や周知表示を保護し、objective の『どのような行為が違法か』に対応する。続く使用許諾契約群は、著作権で保護されたソフトウェアの利用を契約で許諾する仕組みで、権利の体系(u01)を前提とする。法的保護→契約による利用許諾という流れで概念依存を満たす。前提=u01。5 term。</sub>
用語順: 不正競争防止法(13) → 使用許諾契約(5) → ボリュームライセンス契約(3) → サイトライセンス契約(0) → CAL(0)

### strategy-02-04-u03 ソフトウェアの提供形態と利用形態　[5語 / topic内題数計6] （前提: strategy-02-04-u01, strategy-02-04-u02）
オープンソース・フリー・パブリックドメインといった権利の付与範囲が異なる提供形態と、アクティベーション・サブスクリプションという利用制御・提供方式を理解する。
<sub>分割根拠: u01(権利の体系)・u02(契約による許諾)を踏まえ、ソフトウェアが実際にどのような形態で提供され、どこまで自由に使えるかを学ぶ最終ユニット。OSS→フリー→パブリックドメインと、付与される権利・自由度が広がる順に並べ、続いて現代的な利用制御(アクティベーション)と提供方式(サブスクリプション)を置く。許諾(u02)の知識が提供形態ごとの権利範囲の理解を支えるため、前提=u01,u02。5 term。</sub>
用語順: オープンソースソフトウェア(0) → フリーソフトウェア(2) → パブリックドメインソフトウェア(1) → アクティベーション(2) → サブスクリプション(1)

## management-11-29 — サービスマネジメントシステム（management）
- 目標: サービスマネジメントシステムの概要やサービスデスクなどの関連項目を理解する。

### management-11-29-u01 サービスマネジメントの基盤とサービスレベル合意　[5語 / topic内題数計27] （前提なし）
SMS の全体像と、提供するサービスを定義し利用者と合意する仕組み(要求事項・サービスレベル管理・サービスカタログ)を理解する。
<sub>分割根拠: 他の全プロセスの土台となる『何を/どの水準で提供するかを合意・定義する層』を最初に束ねた。SMS(全体枠組み)→ サービスの要求事項(満たすべき条件)→ サービスレベル管理/SLM(水準の合意と維持)→ サービスカタログ(合意済みサービスの一覧化)という定義系の概念で一貫している。後続の運用・移行・改善すべての前提となるため位置は冒頭、prerequisites なし。5 term で D-115 範囲内。</sub>
用語順: サービスマネジメントシステム(1) → サービスの要求事項(0) → サービスレベル管理(14) → SLM(11) → サービスカタログ(1)

### management-11-29-u02 需要への対応と運用プロセス　[5語 / topic内題数計59] （前提: management-11-29-u01）
サービスへの需要・利用者要求の受付から、インシデント・問題への対処、上位への引き継ぎ(エスカレーション)まで、日常運用の中核プロセスを理解する。
<sub>分割根拠: 合意した水準を実際に維持するための『日々の需要受付と障害対処』の運用フローを一つの概念単位に束ねた。需要管理(需要の予測・調整)とサービス要求管理(個別要求の受付)が運用への入口、インシデント管理(暫定復旧)→ 問題管理(恒久対策)が対処の流れ、エスカレーションが全プロセスを横断する上位引き継ぎ機構。前ラウンドで u02 が 4 term だった D-115 逸脱を是正するため、需要管理 を u01 から本 unit へ概念的に移動し 5 term とした(需要管理 とサービス要求管理 は同じ需要・要求の入口グループ)。合意層(u01)を前提とするため位置は 2 番目。</sub>
用語順: 需要管理(0) → サービス要求管理(4) → インシデント管理(33) → 問題管理(16) → エスカレーション(6)

### management-11-29-u03 変更とサービスの移行　[5語 / topic内題数計37] （前提: management-11-29-u01, management-11-29-u02）
構成情報の管理を土台に、変更管理・リリース及び展開管理を経て、新規・変更後のサービスを本番へ移行する一連のプロセスを理解する。
<sub>分割根拠: サービスへ変更を加え本番環境へ送り出す『変更・移行系』を一つの概念単位に束ねた。構成管理(対象情報の正確な把握)→ 変更管理(変更の承認・統制)→ リリース及び展開管理(実装・展開)→ サービスの移行(本番移行)→ サービス受入れ基準(移行可否の判定基準)という、変更が本番に至るまでの依存連鎖で一貫。日常運用(u02)が回っている前提で変更を扱うため位置は 3 番目。5 term で D-115 範囲内。</sub>
用語順: 構成管理(8) → 変更管理(14) → リリース及び展開管理(14) → サービスの移行(1) → サービス受入れ基準(0)

### management-11-29-u04 サービスの維持と継続的改善　[5語 / topic内題数計27] （前提: management-11-29-u02, management-11-29-u03）
可用性・継続性の確保とサービスの報告を通じて状態を把握し、PDCA による継続的改善でサービス品質を高め続ける仕組みを理解する。
<sub>分割根拠: 運用・移行で提供されるサービスを『安定維持し、測定・報告に基づき改善し続ける』管理層を束ねた。サービス可用性管理(止めない)→ サービス継続管理(災害時にも継続)→ サービスの報告(状態の可視化)→ 継続的改善(報告を踏まえた改善活動)→ PDCA(改善を回す方法論)という、維持から改善方法論への概念依存で一貫。運用・移行(u02・u03)の成果を前提に改善するため位置は 4 番目。5 term で D-115 範囲内。</sub>
用語順: サービス可用性管理(8) → サービス継続管理(1) → サービスの報告(3) → 継続的改善(8) → PDCA(7)

### management-11-29-u05 サービスデスクと利用者支援ツール　[5語 / topic内題数計33] （前提: management-11-29-u02）
利用者との単一窓口であるサービスデスクの役割(SPOC)と、FAQ・チャットボット・AIOps など対応を効率化する支援手段を理解する。
<sub>分割根拠: objective が明示する『サービスデスクなどの関連項目』を、利用者接点と支援ツールの概念単位として束ねた。サービスデスク(窓口本体)→ SPOC(単一窓口という原則)→ FAQ → チャットボット → AIOps という『窓口→原則→支援ツール(基本→AI 活用)』の概念チェーンで一貫。運用プロセス(u02)を理解した上で利用者接点を学ぶと役割が明確になるため最後に配置。5 term で D-115 範囲内。</sub>

<sub>分割根拠: デュアル/デュプレックス/クラスタ/レプリケーション/RAID/NAS は、いずれも『複数の構成要素を組んで可用性・信頼性を高める』という共通テーマで束ねられる。複数台を組むには u01 の処理形態(分散など)の理解が前提のため2番目。シラバス上離れた RAID(19)・NAS(18) も機械順でなくストレージの信頼性という意味で同ユニットに統合した。term 数6で 5〜8 を満たす。</sub>

### technology-16-43-u03 仮想化の基本と方式　[6語 / topic内題数計12] （前提: technology-16-43-u01）
物理資源を論理的に分割・統合する仮想化の考え方と、その実現方式(ホスト型/ハイパーバイザー型/コンテナ型)および仮想マシンの概念を学ぶ。
<sub>分割根拠: 前案で 10 語に膨張し self-audit も誤記していた仮想化群を、概念の核(仮想化の定義・実現方式 3 種・仮想マシン)に絞った 6 語ユニットへ再分割した(Rule D 是正)。仮想化の定義→方式 3 種→生成される実体(VM/仮想マシン)の流れで構成。応用・運用(VDI/シンクライアント/マイグレーション系)は u04 へ分離。仮想化は物理構成(u01/u02)を抽象化する発展技術のため、それらの後に配置。term 数6で 5〜8 を満たす。</sub>
