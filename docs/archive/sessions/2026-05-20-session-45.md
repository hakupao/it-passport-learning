## 2. Turn 2 — Context7 docs audit: @vercel/kv deprecation pivot

Per D-019 §3a "你来定" obligation, consulted authoritative docs before locking D-100:

### Context7 query #1: `/vercel/storage` (High rep, 59 snippets, 71.79 score)

**Finding** — quoted verbatim from `https://github.com/vercel/storage/blob/main/README.md`:

> "The packages **@vercel/postgres and @vercel/kv are deprecated**, and the associated products are no longer supported. Users can find alternative storage solutions through the Vercel Marketplace as native integrations."
>
> "For Postgres, users can transition to Neon database solutions, while **for KV, the recommended path is to use Upstash Redis**."

### Context7 query #2: `/websites/upstash_redis` (High rep, 3280 snippets, 79.94 score)

Verified current `@upstash/redis` package API:

```ts
import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv();
// or:
const redis = new Redis({
  url: process.env.UPSTASH_REDIS_REST_URL,
  token: process.env.UPSTASH_REDIS_REST_TOKEN,
});

const count = await redis.incrby("counter", 122);
await redis.expire("counter", 172800);
```

REST-based HTTP client (works in Edge runtime + Node runtime + browser); `fromEnv()` reads standard env vars `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` (set by Vercel Marketplace integration automatically).

### Pivot impact

Q3=a "Vercel KV (Upstash Redis)" remains semantically correct (same Upstash Redis backend), but the package surface changes from the deprecated `@vercel/kv` wrapper to the upstream `@upstash/redis` package. The Vercel Marketplace 1-click integration still provisions an Upstash Redis instance and wires the standard env vars — only the consuming Node package name differs.

**Cost stays $0** for α single-user: Upstash free tier = 10k commands/day + 256 MB. Cap counter generates ≤ 100 commands/day at α traffic (single-user, manual smoke-test only); below free-tier ceiling by 100×.

Behaviour summary:
- Sync logs (`[cap]`, `[cap-wall]`) emit before any await — visibility survives lambda termination
- Async Redis call (`INCRBY` + `EXPIRE`) is fire-and-forgettable from caller side (`void recordCapEvent(...)`)
- `[cap-breach]` log fires only on the call that crosses 5M μUSD (first-crossing detection per LD-3)
- `[cap-redis-error]` log catches Redis throws; routes never crash
- `[cap-mode-stub]` annotation fires when β mode env set but α-silent behaviour applies
