#### Q17 (OQ-21) — 三语字段的承载形态

**背景**: D-009 锁了"每个文本字段 = `{jp, zh, en}`"，但**具体字段在 Pydantic 模型里长什么样**还没定。这一选会影响所有下游 schema 的形态、JSON 体积、Phase 3 前端访问写法、SQLite 表结构。

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 嵌套** | `title: {jp: "...", zh: "...", en: "..."}` | 结构清晰；新加文本字段只改 1 处；JSON Schema 紧凑 | 访问要 `term.title.jp` 多一层；SQLite 映射要拍平 |
| **(B) 扁平** | `title_jp / title_zh / title_en` | 直接 `term.title_jp`；SQLite 一对一映射 | 字段数 ×3 膨胀；新加 1 个文本字段要改 3 处；schema 视觉噪声大 |
| **(C) 拆 record** | `translations: [{lang: "jp", text: "..."}, {lang: "zh", text: "..."}, ...]` | RDB-friendly（一张 translations 表）；新增语言无需改 schema | 访问罗嗦（要 list filter）；JSON 噪声重；不天然支持类型校验"必有 jp/zh/en 三条" |

**关键 trade-off**: Phase 3 前端消费友好度 ↔ schema 体积 ↔ SQLite 表结构

**附议**: 如果选 (A) 嵌套，是否考虑给嵌套结构起一个复用类型（如 `Trilingual`）以避免重复声明？这会再决定下一层："`Trilingual` 是 `BaseModel` 还是 `TypedDict` 还是 `dataclass`"。

---

#### Q18 (OQ-22) — 翻译未到位时的语言字段语义

**背景**: D-008 pipeline 是 stage 0~7，**翻译发生在 stage 5**。即 stage 0~4 抽出来的实体只有 jp 文本，stage 5 之后才三语齐全。问题: **中间各 stage 的 schema 怎么表达"zh/en 还没填"？**

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 占位符** | 三字段都必填；未翻译时填 `"<UNTRANSLATED>"` 等哨兵值 | schema 单一不分裂；类型校验始终 `str` not `Optional[str]` | 哨兵值容易"漏过滤"进 final output；运行时要全程检查 |
| **(B) Optional** | `zh: Optional[str] = None`；final output 阶段统一断言 not-None | Pythonic，类型系统直接表达"未翻译"；`None` 不会"假装翻译过" | schema 在中间 stage 不严格；`None` 流到下游容易 NPE |
| **(C) 阶段化 schema** | 用不同 Pydantic 模型: `RawTerm(jp only)` → `TranslatedTerm(jp+zh+en)` | 每 stage 严格 self-contained；类型系统强制 stage 顺序 | schema 数量乘以 stage 数 (`Raw*` / `Cleaned*` / `Translated*`...)；维护成本高 |

**关键 trade-off**: schema 数量 ↔ 字段空值灵活度 ↔ 终态严格度

**附议**: 这一选与 Q20 (stage 契约校验颗粒度) 强耦合 —— 如果 Q20 选"全 stage 强校验"，则 Q18 几乎被迫选 (C) 阶段化 schema；反之 Q20 选"只终点校验"则 Q18 可走 (B) Optional。建议两题一起看。

#### Q19 (OQ-23) — Entity 类型机制

**背景**: 第一批至少 6 类 entity: `chapter / section / term / question / table / figure`（per D-008 + D-022）。但 D-010 (cert-agnostic) 暗示未来 cert 插件可能加自定义 type（如法律资格的"判例"、医师资格的"病例"）。Pydantic 怎么表达这个"基础类型固定 + 扩展类型可加"？

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) Closed Literal + Discriminated Union** | `type: Literal["chapter","section","term","question","table","figure"]`，配合 Pydantic v2 `Field(discriminator="type")` | 强类型；IDE 自动补全；JSON Schema 漂亮 | 加新 type 必须改库；cert 插件无法自带新 entity type |
| **(B) Open via 插件注册** | 插件可 `@register_entity_type("custom")`；type 字段是普通 `str` | 完全 cert-agnostic；第三方扩展无门槛 | 失去 closed schema 的所有保证；下游要靠 if-else 处理未知 type |
| **(C) 折中: Core + Extension Namespace** | 库内置 closed core 6 类（如 (A)），插件加自定义类型必须用 namespace 前缀（如 `ext::case_law`）走宽松 dict 字段 `extension_data` | core 路径强类型；扩展路径不破坏 core；JSON Schema 仍可表达 core | 实现复杂；user 学习成本（"core type" vs "ext type"两套写法） |

**关键 trade-off**: 类型严格度 ↔ cert-agnostic 扩展性 ↔ 实现 + 文档复杂度

**附议**: 这一题也涉及 D-026 (插件版本化机制) —— 如果选 (B) 或 (C)，第三方插件加新 entity type 时要不要 schema 版本号也跟着变？涉及 Q21。

#### Q20 (OQ-24) — Stage 间契约校验颗粒度

**背景**: D-008 pipeline 有 8 个 stage（含 stage 4.5 glossary）。每两个 stage 之间都有产物 IO（`/data/<cert_id>/runs/<run_id>/<stage>/`）。问题: **Pydantic 校验在哪些 stage 边界生效？**

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 全 stage 强校验** | 每个 stage 入参 + 出参都过 Pydantic `model_validate`，失败 raise | 错误第一时间捕获；中间产物始终合规 | IO 成本翻倍（每过一道 stage 就 parse + validate 一次）；schema 数量爆炸（per Q18) |
| **(B) 终点校验** | 中间 stage 用轻量 dataclass / TypedDict，只在 stage 7 (export) 前过 Pydantic | 运行成本低；schema 数量少 | 中间 stage 出错难定位；很可能等到 export 才暴露问题 |
| **(C) 关键边界校验** | 只在跨"语义层级跳变"的 stage 边界严校验，例如:<br>• OCR → Structure（非结构化 → 结构化）<br>• Structure → Translate（jp → 三语）<br>• Translate → Export（内部 → 最终发布）<br>其他 stage 用轻量类型 | 兼顾错误捕获和性能；语义边界天然合理 | "关键边界"定义需斟酌（哪些算关键）；实现非纯粹（一会 Pydantic 一会 TypedDict） |

**关键 trade-off**: 错误早期捕获 ↔ IO 性能 ↔ 实现复杂度

**附议**: 这题与 Q18 (中间 stage 字段空值语义) 强耦合，建议合并讨论。

#### Q21 (OQ-25) — Schema version 演进策略

**背景**: cert-extractor 是要**长期演化**的库（路线图到 Phase 5 = 通用框架开放给第三方 cert）。schema 必然会演化（新加字段 / 改语义 / 删旧字段）。问题: **怎么标识版本，让下游消费方（Phase 3 前端、第三方插件）能反向兼容？**

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 顶层 dataset 版本** | 输出 JSON 顶部加 `schema_version: "1.0.0"` (SemVer)；per-entity 不写 | schema 体积小；版本号集中管理；和库 SemVer 解耦 | record 级别没法独立溯源；如果一个 dataset 内混合多版本（很少见但可能）会乱 |
| **(B) per-entity 版本** | 每个 entity record 都带 `schema_version` | record-level 反向兼容；理论最稳 | schema 体积膨胀（重复版本号）；维护成本高 |
| **(C) 不显式写版本** | 完全靠 git tag / 库 PyPI version 兜底；schema 不写版本 | 最少噪声；零 schema 维护 | 第三方消费者要先知道是哪版库导出的（外部信息源依赖）；离线 JSON 文件单独漂移就无法识别 |

**关键 trade-off**: 反向兼容成本 ↔ schema 体积 ↔ 第三方消费者理解负担

**附议**: 这一题与 D-026 (插件版本声明 `__cert_extractor_min_version__`) 形成"双版本号"的可能 —— 库 version + schema version。是否需要同步？还是脱钩？这是子问题。

### 4.1 Q17 (OQ-21) ✅ — 三语字段承载形态 → A 嵌套

**议题边界回顾**: D-009 锁的是"每个文本字段 = `{jp, zh, en}`" 这条原则；Pydantic 字段层面具体怎么排没定。

**User 答**: A（嵌套）

**Claude 决策推理（含 sub-decision）**:

A 模式下绕不开的下一层："`Trilingual` 容器形态选什么？" 三个候选:

| 容器 | 是否选 | 理由 |
|---|---|---|
| **Pydantic `BaseModel`** | ✅ **选** | 与 Q19=A (Discriminated Union) + Q20=A (全 stage 强校验) 一致；自动获得 `model_validate` / `model_json_schema` / 校验器；零额外设计成本 |
| `TypedDict` | ❌ | 不能挂 validator；不能直接进 Discriminated Union；与 Q20 矛盾 |
| `dataclass` | ❌ | 同 TypedDict 短板，且 Pydantic v2 对 dataclass 的支持是"加层皮"，不如直接 BaseModel |

→ 锁 sub-decision: `Trilingual = BaseModel`。

**锁定 D-054 spec**:

```python
# packages/extractor/src/cert_extractor/schema/common.py（实施期参照，本场不写）
from pydantic import BaseModel, ConfigDict, Field

class Trilingual(BaseModel):
    """三语文本载体。所有日/中/英文本字段统一走此类型。"""
    model_config = ConfigDict(extra="forbid", strict=True)

    jp: str = Field(..., min_length=1, description="日语原文（source 语言，stage 0~4 即填）")
    zh: str = Field(..., min_length=1, description="中文翻译（stage 5 之前可填占位符 D-055）")
    en: str = Field(..., min_length=1, description="英文翻译（stage 5 之前可填占位符 D-055）")
```

字段命名: `jp / zh / en`（短，与项目内现有约定一致；不用 BCP-47 `ja / zh-CN / en` 因为字段名不能带连字符且与 source code 风格不符）。

**Trade-off 备查**:

| 维度 | A 嵌套 (选) | B 扁平 (拒) | C 拆 record (拒) |
|---|---|---|---|
| Phase 3 前端访问 | `term.title.jp` 多一层 | `term.title_jp` 直接 | `term.translations.find(t => t.lang === 'jp')` 罗嗦 |
| 加新文本字段 | 改 1 处声明 | 改 3 处 | 改 1 处但要校验 list 完整性 |
| schema 体积 | 中 | 大（×3 字段） | 小但 list 噪声 |
| SQLite 映射 | 拍平: `xxx_jp/xxx_zh/xxx_en` 列 | 一对一 | 单独 translations 表（join 成本） |
| 与 Q19/Q20 协调 | ✅ Pydantic 原生 | ✅ 但啰嗦 | ✅ 但 Discriminated Union 写起来怪 |

**Rejected alternatives + reasons**:

- **B 扁平**: 加 1 个文本字段要改 3 处 + schema 视觉噪声大，违背"长期演化"原则。
- **C 拆 record**: 不天然校验"必有 jp/zh/en 三条"（要写 model_validator 兜底），且与"翻译占位符"语义难协调（list 里塞 None / 占位符更怪）。

### 4.2 Q18 (OQ-22) ✅ — 翻译未到位语义 → A 占位符

**议题边界回顾**: pipeline stage 0~4 只有 jp，stage 5 后才三语齐全；中间态怎么填 zh/en？

**User 答**: A（占位符）

**Claude 决策推理（含 sub-decision）**:

A 模式下要决两个细节:

1. **哨兵字符串选什么？** 候选: `"<UNTRANSLATED>"` / `"__UNTRANSLATED__"` / `"#UNTRANSLATED"` / 空字符串 `""`。
   - 选 **`"<UNTRANSLATED>"`**: 尖括号确保不会被误认为自然语言（任何语言都不会以 `<` 开头一段词）；大写 + 全词使 grep / regex 识别 100% 准确。
   - 拒空字符串: 与"未填"语义混；Q17 的 `Field(min_length=1)` 也不允许空串。
   - 拒 `__xxx__`: Python dunder 风格，会与 LLM 输出（可能带下划线）的天然词混淆。

2. **stage 7 export 前的 contract 怎么落？** 在 export 阶段的 schema 上挂 model_validator 强制 assert：

```python
# 实施期参照
UNTRANSLATED = "<UNTRANSLATED>"

class FinalDataset(BaseModel):
    """Stage 7 export 形态 — 不允许任何字段含哨兵。"""
    items: list[Entity]

    @model_validator(mode="after")
    def no_untranslated_leakage(self):
        for item in self.items:
            for t in _walk_trilingual_fields(item):
                for lang in ("jp", "zh", "en"):
                    if getattr(t, lang) == UNTRANSLATED:
                        raise ValueError(f"Untranslated leak in {item.id}: {lang}")
        return self
```

**锁定 D-055 spec**:

- 常量: `UNTRANSLATED = "<UNTRANSLATED>"`，定义在 `cert_extractor.schema.common` 模块顶层
- 中间 stage（0~5 之前）允许 `Trilingual.zh / .en` 填 `UNTRANSLATED`
- Stage 5 翻译完成后必须填真实文本
- Stage 7 export 前 `FinalDataset` model 强制 validator: 任何 `UNTRANSLATED` 残留 → ValidationError → raise → 归档到 `failures/`

**Trade-off 备查**:

| 维度 | A 占位符 (选) | B Optional (拒) | C 阶段化 schema (拒) |
|---|---|---|---|
| schema 数量 | **1 套（最少）** | 1 套 | N 套（每 stage 一套） |
| 字段类型一致 | ✅ 全程 `str` | ❌ `Optional[str]` 终态需 cast | ✅ 各阶段内一致 |
| 中间 stage 错误传播风险 | 低（哨兵显眼） | 中（None 流到下游 NPE） | 极低（schema 强制） |
| 终态严格度 | 高（model_validator 兜底） | 中（要后处理 cast） | 高 |
| 实现 + 维护成本 | **低** | 中 | **高（schema 数 ×N）** |
| 与 Q20=A 协调 | ✅ 完美 | ⚠️ 中间 stage Optional 削弱了"全 stage 强校验"意义 | ✅ 但成本高 |

**Rejected alternatives + reasons**:

- **B Optional**: 与 Q20=A 强校验冲突（Optional 弱化了类型保证），且 `None` 比 `"<UNTRANSLATED>"` 更难 grep 定位。
- **C 阶段化 schema**: 维护成本爆炸（8 stage 每个都要全套 schema），违背"最简" 原则；user 全选 A 也表达了对单 schema 的偏好。

### 4.3 Q19 (OQ-23) ✅ — Entity 类型机制 → A Closed

**议题边界回顾**: 6 类 entity (chapter/section/term/question/table/figure) 是固定的还是可扩展的？

**User 答**: A（Closed Literal + Discriminated Union）

**Claude 决策推理**:

按 Pydantic v2 官方推荐，Discriminated Union 是 polymorphic schema 的标配：

```python
from typing import Literal, Annotated
from pydantic import BaseModel, Field

class _BaseEntity(BaseModel):
    id: str
    page: int          # D-022 Hybrid 锚点
    block_id: str      # D-022
    section_path: list[str]  # D-022

class Chapter(_BaseEntity):
    type: Literal["chapter"] = "chapter"
    title: Trilingual

class Section(_BaseEntity):
    type: Literal["section"] = "section"
    title: Trilingual

class Term(_BaseEntity):
    type: Literal["term"] = "term"
    surface: Trilingual
    definition: Trilingual
    kana_helper: KanaHelper | None = None  # D-012

class Question(_BaseEntity):
    type: Literal["question"] = "question"
    stem: Trilingual
    choices: list[Trilingual]
    answer_index: int

class Table(_BaseEntity):
    type: Literal["table"] = "table"
    caption: Trilingual
    rows: list[list[Trilingual]]

class Figure(_BaseEntity):
    type: Literal["figure"] = "figure"
    caption: Trilingual
    image_ref: str  # 指向 /data/.../raw/pages/page_NNN.jpg 的相对路径

Entity = Annotated[
    Chapter | Section | Term | Question | Table | Figure,
    Field(discriminator="type"),
]
```

**Phase 5 扩展机制**: 留待真正出现"第二种 cert"时再决定（例如法律资格的"判例" / 医师资格的"病例"）。届时可考虑选项 (C) 折中方案，但**不预设**。

**锁定 D-056 spec**:

- Core 6 entity types 锁死: `chapter / section / term / question / table / figure`
- 实现: Pydantic v2 `Annotated[Union[...], Field(discriminator="type")]`
- 每个 entity model 都继承 `_BaseEntity`（含 D-022 Hybrid 锚点 4 字段）
- 不开放运行时 `@register_entity_type` 机制
- Phase 5 再议扩展（届时可能反向 promote 到 D-056-v2）

**Trade-off 备查**:

| 维度 | A Closed (选) | B Open (拒) | C 折中 (拒) |
|---|---|---|---|
| 类型严格度 | **高（IDE / mypy / runtime 全覆盖）** | 低（type 是普通 str） | 中（core 强 / ext 弱） |
| cert-agnostic 扩展性 | 低（Phase 5 前不可扩） | 高 | 中 |
| 实现复杂度 | **低** | 中（要写 registry） | 高（双路径） |
| 第三方插件友好度 | 低 | 高 | 中 |
| 当前 Phase 1 实际需要 | ✅ 只需 itpassport_r6，不需扩展 | over-engineered | over-engineered |

**Rejected alternatives + reasons**:

- **B Open**: 当前 Phase 1 只跑一本书一个 cert，提前开放扩展机制是 over-engineering；失去 closed schema 的所有保证（IDE / mypy / runtime）。
- **C 折中**: Phase 1 没有第二个 cert 验证 namespace 设计的必要性，等真实需求出现再设计；当前预设 namespace = pre-mature abstraction。

### 4.4 Q20 (OQ-24) ✅ — Stage 间契约校验颗粒度 → A 全 stage 强校验

**议题边界回顾**: 8 个 stage 都校验？还是只在终点？还是关键边界？

**User 答**: A（全 stage 强校验）

**Claude 决策推理（含 sub-decisions）**:

A 模式下要决三个细节:

1. **每 stage 校验时机**: 
   - 入参（读上一 stage 产物）→ 出参（写本 stage 产物）都过 `model_validate`
   - 不允许 `model_construct`（跳过 validation 的快速路径）—— 与"最严"原则冲突
2. **失败处理**:
   - `ValidationError` → raise → pipeline halt
   - 同时归档 `failures/step_NN_attempt_X.md`（per 规则 B）
   - 含 input snapshot + 错误堆栈 + 时间戳
3. **性能权衡**:
   - IO 翻倍（每 stage 双向 validate）= 接受成本
   - 加速选项（如 batch validate）留待**真出现性能问题**时再 profile，**不预优化**

**锁定 D-057 spec**:

- 8 stage 中每个 stage 都按"入参 validate → 处理 → 出参 validate → 写盘 → 下 stage 入参 validate"循环
- 全程使用 `Model.model_validate(data)` 而非 `model_construct(...)`
- Pydantic config 全局: `model_config = ConfigDict(extra="forbid", strict=True, frozen=False)`
- Validation failure → raise + 写 `failures/` + halt pipeline（不 swallow）
- 每 stage 入/出 schema 不允许"silent default"（如 `Optional[str] = None` 要明确 + 有理由 + 在 docstring 说明）

**Trade-off 备查**:

| 维度 | A 全 stage (选) | B 终点校验 (拒) | C 关键边界 (拒) |
|---|---|---|---|
| 错误捕获时机 | **最早** | 最晚 | 中间 |
| IO 性能 | 翻倍 | **最快** | 中 |
| 中间产物可信度 | **极高** | 低 | 中 |
| 实现一致性 | ✅ 单 pattern | ✅ 单 pattern | ⚠️ 双 pattern 易乱 |
| 调试友好度 | **极高（错误第一时间暴露）** | 低（错误堆积到终点才炸） | 中 |

**Rejected alternatives + reasons**:

- **B 终点校验**: 中间错误堆积，pipeline 跑 5 stage 后炸，定位成本极高；与 Tier 3 evidence 文化冲突。
- **C 关键边界**: 双 pattern（强 vs 轻）增加心智负担和维护成本；"哪个边界算关键"需持续维护一份名单，反而更乱。

### 4.5 Q21 (OQ-25) ✅ — Schema version 演进策略 → A 顶层 envelope（**Claude 决**）

**议题边界回顾**: cert-extractor 是要长期演化的库，schema 版本号怎么标？

**User 委托原话**:

> "21-我看不懂，你按照最优解帮我决定"

按 CLAUDE.md "你来定" 协议（D-019）:

1. ✅ 查权威 docs（不靠记忆）
2. ✅ 锁 D + 显式理由
3. ✅ 列 rejected alternatives
4. ✅ 同回合写回 session log + STATE.md

#### Claude 查 docs 过程

通过 context7 拉取 Pydantic 官方 docs（`/websites/pydantic_dev_validation`），关键发现:

- Pydantic v2 **不强制** schema versioning pattern（这是应用层决定）
- 官方提供 `model_config = ConfigDict(json_schema_extra={...})` 注入 schema metadata
- 官方提供 `model_json_schema()` 导出 JSON Schema

业界共识参照（不靠 Pydantic 一家言）:

| 标准 | 版本号位置 | 形态 |
|---|---|---|
| **JSON Schema spec** | 顶层 `$schema` URL | `"https://json-schema.org/draft/2020-12/schema"` |
| **OpenAPI spec** | 顶层 `openapi` | `"openapi": "3.1.0"` |
| **package.json** | 顶层 `"version"` | SemVer 字符串 |
| **pyproject.toml** | 顶层 `[project] version` | SemVer |
| **Cargo.toml** | 顶层 `[package] version` | SemVer |

→ "顶层 + SemVer" 是跨语言、跨工具链的事实标准。选项 (A) 与之对齐。

#### Claude 决策（**最优解 = A 的细化版**）

(A) 是基础正解但不够具体。"只一个 `schema_version: '1.0.0'`" 不足以支撑长期演化和第三方消费。需要完整 envelope:

```json
{
  "schema_version": "1.0.0",
  "extractor_version": "0.1.0",
  "cert_id": "itpassport_r6",
  "generated_at": "2026-05-06T13:24:55+09:00",
  "items": [ /* Entity[] */ ]
}
```

**4 字段的理由**:

| 字段 | 必要性 | 用途 |
|---|---|---|
| `schema_version` | **必** | 下游消费方判断兼容性的唯一权威依据；与库版本解耦（库可以 patch 而 schema 不变） |
| `extractor_version` | 强烈推荐 | 调试 / 排查 bug 时知道是哪个库版本生成的 |
| `cert_id` | **必** | D-010 的 cert-agnostic 顶层标识；下游可直接路由不同 cert 处理逻辑 |
| `generated_at` | 强烈推荐 | ISO 8601；增量更新 / 缓存判定 / 审计追溯 |

#### SemVer bump 规则（D-058 附则）

| 变更类型 | bump | 示例 |
|---|---|---|
| **MAJOR** | x.0.0 → (x+1).0.0 | 删字段 / 改字段类型 / 改字段语义 / 删 entity type / Discriminated Union value 改名 |
| **MINOR** | x.y.0 → x.(y+1).0 | 加 optional 字段 / 加 entity type / 加 enum value / 加新的 entity property |
| **PATCH** | x.y.z → x.y.(z+1) | 仅 docstring / description 文本变更 / bug fix（不影响 schema 形态） |

> **注**: schema_version 与库 PyPI version (`extractor_version`) **解耦**。库可以 0.1.0 → 0.1.1 (PATCH) 时 schema 不变（保持 1.0.0）；反之 schema 升 2.0.0 时库可能跳 1.0.0 (MAJOR 联动也常见)。

#### 锁定 D-058 spec

- Stage 7 export 输出的所有 JSON 文件顶层 envelope 必须含 4 字段: `schema_version / extractor_version / cert_id / generated_at`
- `schema_version` 走 SemVer，bump 规则如上表
- 实现位置: `cert_extractor.schema.export.FinalDataset` model
- `extractor_version` 从 `importlib.metadata.version("cert-extractor")` 拉取
- `generated_at` 用 `datetime.now(tz=ZoneInfo("Asia/Tokyo"))` ISO 8601 字符串
- 当 schema 演化触发 MAJOR bump 时，必须同回合写一条 ADR (`docs/decisions/D-NNN-schema-bump-vN.md`)

**Trade-off 备查**:

| 维度 | A 顶层 envelope (选) | B per-entity (拒) | C 不写 (拒) |
|---|---|---|---|
| 反向兼容工具链支持 | ✅ 顶层一目了然 | ⚠️ 工具要 walk 所有 entity | ❌ 离线无识别 |
| schema 体积 | 增量 4 字段（终点 1 次） | ×N entity 数 | 0 |
| 第三方消费者理解负担 | **低** | 中 | **高（要外查）** |
| 业界对齐度 | **高（OpenAPI / JSON Schema / package.json 同模式）** | 低（很少这么干） | 最低 |
| 多版本 dataset 混合场景 | 一份 dataset 一个版本（合理预设） | 支持但当前不需要 | 不支持 |

**Rejected alternatives + reasons**:

- **B per-entity 版本**: 当前一份 dataset = 一次 extractor 运行 = 同版本，per-entity 版本 100% 重复，schema 体积浪费；除非未来支持"多次运行 merge"才有价值。
- **C 不写版本**: 离线 JSON 文件流通时无法识别版本（如用户下载了一个 release JSON 隔月再用），下游消费方需"猜"或"问"，违反 self-describing 原则。
