#### Q32 (OQ-35) — Anthropic 接入方式

**背景**: User max 20 plan ($200/月) 已就绪（OQ-06）。但 max plan 是订阅模式（OAuth），不是 pay-as-you-go API key。Claude Agent SDK (per context7 验证) 支持读 OAuth credentials，但有约束:

- 必须用 `claude-agent-sdk` (Python)，不能用 `anthropic` SDK 直接
- Max plan 有 5h rate limit / quota（Claude Code 默认 quota）
- 大批量 OCR + Translation 任务可能撞 quota（需在 5h 窗口分批）

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) Agent SDK + max 20 plan** (推荐基线) | cert-extractor 通过 `claude-agent-sdk` 走 OAuth | "零额外 LLM 费用"（已订阅） | 撞 quota 时要等 5h reset；批量任务需分片 |
| **(B) 独立 ANTHROPIC_API_KEY** (pay-as-you-go) | 单独申请 API key + 充值 | 不限 quota，可大批量并行 | 额外开销（500 pages × 翻译 = 估 $20–$50；不算太多但是新支出） |
| **(C) 双轨**: dev / 抽检走 (A) / 全本 batch 跑走 (B) | dev 用 OAuth 省钱，prod 大跑用 API key 不撞 quota | 钱 + 速度兼顾 | 实施复杂度高（双 SDK 路径，规则 D 风格的"双 path"反模式） |

**关键 trade-off**: 钱 ↔ quota 限制 / 速度 ↔ 实施复杂度

**Claude 推荐**: **(A) 起手** — Phase 1 第一次跑就用 max 20 plan，撞 quota 就分多批次。如果发现实际撞 quota 严重影响速度，再升级到 (C) 双轨。这符合 user "前提是做出来" 偏好（最低实施成本起步）。

**候选**:

| 选项 | 形态 | 优点 | 代价 |

### 4.2 Q33 (OQ-36) ✅ — Mistral 接入 + dry-run → B SDK + 50 pages dry-run（**Claude 决**）

**Dry-run 流程**:

1. 单 chapter 跑 stage 0+1（unpack + Mistral OCR）≈ 50 pages
2. 输出 `data/<cert>/runs/dry_run_<ts>/ocr/` + `evidence/.../step_01_audit.md`
3. **暂停 + user review**（per D-063 WARN 等人决风格）
4. PASS → 全本 stage 0+1
5. FAIL → 调 Mistral 配置 / 切 Hard Re-OCR / 重 dry-run

**锁定 D-070 spec**:

- 接入: `mistralai` Python SDK (官方)
- Dry-run 范围: 单 chapter / ~50 pages / stage 0+1 only

- B 多 cap = 多维兜底但缺 WARN 灰色区间
- **C 软硬 cap (三档)** = 与 D-063 PASS/WARN/FAIL 完美对齐

**软硬 cap 阈值**（默认值，YAML 可调）:

| 维度 | 软 cap (WARN) | 硬 cap (FAIL → halt) |
|---|---|---|
| Wall-time | 2h | 8h |
| 累计 Mistral USD | $5 | $20 |
| 累计 Anthropic USD (仅 API key 路径) | $5 | $30 |
| 累计 fail 次数 (任意 stage) | 10 | 30 |

软 cap 触发: log warning + 写 `failures/<run>/SOFT_CAP_WARN.md` + **暂停等 user 决定**
硬 cap 触发: 自动 halt + 写 `failures/<run>/EMERGENCY_HALT.md` + 触发规则 C mini-retro

**与 D-063 关系**: D-063 是 stage 级抽检 FAIL，D-071 是 run 级总预算熔断；层级不同但风格一致。

**阈值数值依据**:
- Wall-time 2h soft / 8h hard: max 20 plan 5h quota window；2h 给 quota buffer，8h 跨 quota window 不可行
- Cost $5 soft / $20 hard: 全本预估 ~$0.50 → 10x 早期警告，40x 彻底失控
- Fail 10 / 30: D-063 max 3 retries × 5 stages ≈ 15 fail 重试上限，10 软警告 systemic bug，30 硬熔断

**锁定 D-071 spec**:

- 软硬 cap 阈值如上表
- 实施位置: `cert_extractor.budget.BudgetMonitor`（实施期写）
- 触发: 每 stage 完成后 `BudgetMonitor.check()`
- 软 cap WARN: 写 `SOFT_CAP_WARN.md` + halt 等 user
- 硬 cap FAIL: 写 `EMERGENCY_HALT.md` + halt 自动 + 触发规则 C
- pipeline YAML override (但 hard cap 不可超 soft × 5，防误关)
- 落独立 ADR `D-071-budget-cap-and-emergency-halt.md`

**Trade-off 备查**:

| 维度 | A 单 cap (拒) | B 多 cap 单档 (拒) | C 软硬 (选) |
|---|---|---|---|
| 维度覆盖 | 1 (cost) | 多 | 多 |
| 与 D-063 风格 | ❌ | ⚠️ 部分 | ✅ |
| 半自动 (人决) | 无 | 单档 | 三档 |

**Rejected reasons**:
- A: 不抓 wall-time → max plan quota 撞死无熔断
- B: 缺 WARN 灰色区间，违反 "不死板"

### 4.4 Q35 (OQ-38) ✅ — cost tracking → C 双层（**Claude 决**）

**Claude 推理**:

D-062 evidence #16 已留 `cost_estimate` 可选位置。但整 run 总成本要 walk 5 个 evidence 文件相加 → Q34 熔断检测器不便。

- A 仅 evidence cost = 总成本要 walk 5 文件
- B 独立 cost log = 字段冗余
- **C 双层**: per-stage cost 留 evidence (source of truth) + run summary 写 `cost.json` (aggregator)

**cost.json 结构**:

```json
{
  "run_id": "2026-05-06T13-24-55",
  "cert_id": "itpassport_r6",
  "started_at": "2026-05-06T13:24:55+09:00",
  "current": {
    "wall_time_seconds": 4567,
    "mistral_pages": 234,
    "mistral_usd": 0.234,
    "anthropic_tokens_input": 1234567,
    "anthropic_tokens_output": 234567,
    "anthropic_usd": 0.0,
    "fail_count": 3
  },
  "by_stage": {
    "1": {"tokens": 0, "usd": 0.234, "calls": 234},
    "4": {"tokens": 567000, "usd": 0.0, "calls": 12},
    "5": {"tokens": 1234000, "usd": 0.0, "calls": 45},
    "6": {"tokens": 234000, "usd": 0.0, "calls": 18}
  },
  "caps": {
    "soft_warn_at": {"wall_time": 7200, "usd": 5.0, "fails": 10},
    "hard_halt_at": {"wall_time": 28800, "usd": 20.0, "fails": 30}
  }

```

每 stage 完成后:
1. evidence 文件 (D-062 #16) 写本 stage cost
2. cost.json 累加到 `current` + `by_stage[stage]`
3. BudgetMonitor 读 `current` vs `caps`（per D-071）

**锁定 D-072 spec**:

- per-stage cost: 进 evidence (D-062 #16, source of truth)
- run summary cost: 写 `evidence/<cert_id>/runs/<run_id>/cost.json`（aggregator）
- BudgetMonitor reads cost.json
- 不冗余（aggregator vs source 角色分清）
- 不需独立 ADR (字段定义级别决定)

**Rejected reasons**:
- A: 总成本 walk 5 文件，熔断检测器不便
- B: 字段冗余（evidence + cost.json 重复 per-stage 信息）

### 4.5 Q36 (OQ-39) ✅ — Phase 1 启动节奏 → B 单 chapter dry-run + 全本（**Claude 决**）

**Claude 推理**:

按 user "前提是做出来" + Tier 3 evidence 文化 + 规则 C retro:

- A 全本 1-shot = 失败成本最高（500 pages × cost + stage 4 才发现 stage 1 错的风险）
- **B 单 chapter dry-run + 全本** = 平衡速度 + 失败可恢复性
- C 多阶段 dry-run = 节奏太慢（多次 retro 介入）

**三阶段流程**:

```
Stage A: 单 chapter dry-run
   - 1 chapter (~50 pages) 跑完整 stage 0~7 e2e
   - 走全部抽检 (D-060) + reviewer (D-061) + cap (D-071)
   - run_id = dry_run_<timestamp>
   - 输出: data/<cert>/runs/dry_run_<ts>/

Stage B: User retro
   - PASS → Stage C 全本
   - WARN → 调 prompt / 阈值 → 重 Stage A
   - FAIL → 写 RETROSPECTIVE.md (规则 C) → 回讨论

Stage C: 全本 ~500 pages
   - 同 Stage A 流程
   - 输出: data/<cert>/runs/<ts>/
   - 完成 → GitHub Release (D-046)
```

**Dry-run chapter 选择**: 推荐 "中等难度章节"（不是开篇 / 末章），让 dry-run 暴露真实排版复杂度。具体由 user 在 YAML 指定 `dry_run.chapter_index`。

**与已锁规约对接**:

| 规约 | 对接点 |
|---|---|
| 规则 C retro | dry-run FAIL → mini-retro |
| D-063 audit | dry-run 单 chapter 抽检走 PASS/WARN/FAIL 三档 |
| D-071 cap | dry-run + 全本都过软硬 cap |
| D-046 GitHub Release | 全本完成后发版 |

**锁定 D-073 spec**:

- 启动 = 三阶段 (Stage A dry-run / B user retro / C 全本)
- Dry-run run_id 加 `dry_run_` 前缀
- User 必须显式 PASS 后才进 Stage C
- 全本完成 → GitHub Release (per D-046)
- 落独立 ADR `D-073-phase1-launch-strategy.md`（meta 决定 + 跨规则对接）

**Trade-off 备查**:

| 维度 | A 1-shot (拒) | B dry-run+全本 (选) | C 多阶段 (拒) |
|---|---|---|---|
| 失败成本 | 最高 | 中 | 最低 |
| 实施速度 | 最快 | 中 | 慢 |
| User retro 次数 | 0 | 1 | 4+ |
| 与 user "做出来" | ❌ | ✅ | ❌ |

**Rejected reasons**:
- A: 失败成本太高 + stage 4 才发现 stage 1 错风险
- C: 多次 retro 违 "做出来" 偏好；Phase 2+ 真有需求再升级
