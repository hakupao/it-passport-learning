### 4.5 Root-cause analysis of both FAILs

Drilled into both FAIL pages — **neither is an upstream content bug**:

#### FAIL #1: page_042 D5 answer_index_mismatch (safety FAIL → halt)

- **D5 reported**: source has 7 answers `アアアアイイエ` but page has 4 questions → cannot align → safety FAIL
- **Reality**: page_042 cleaned/ does NOT exist (only 4 cleaned/ files total: 002/016/043/047); D5 falls back to ocr/page_042.md
- **Real answer line at L44**: `問題1-1 ア 問題1-2 イ 問題1-3 イ 問題1-4 エ` (4 markers, matches Stage 4 answer_index=[0,1,1,3] exactly)
- **Where the extra 3 `ア` came from**: D5 regex `(?:問題\s*)?\d+\s*[\-...]\s*\d+\s*[\s　]+([アイウエオ])` — because `\s*[\s　]+` allows newlines, it captures choice prefixes like `1-1\nア.` from question stems before the actual answer line
- **The bug is in detectors.py D5 regex, NOT in Stage 4 extraction** (Stage 4 extracted 4 questions correctly)

#### FAIL #2: page_019 D7 numeric_inconsistent

- **D7 reported**: jp/zh have `{54,4,2022,8}`, en has `{54,4,2022}` → conflict → FAIL
- **Reality**: jp/zh `54.4％（2022年4月～2022年8月）` vs en `54.4% (April 2022 – August 2022)` — same content, en spells out month names
- **Same class as Stage A page_014** (`4種類` vs `four types` → WARN); commit `162aebb` severity heuristic only downgrades when populated sets fully agree, not when one set is a strict subset of another
- **The bug is in detectors.py D7 severity heuristic, NOT a Stage 5 translation bug**

Both FAILs are deterministic-detector false positives. Real fix: tighten D5 regex + extend D7 severity heuristic. No LLM API calls needed for the fix path. Re-run audit after fix → expected verdict for both pages = WARN (the legitimate signals remain).
