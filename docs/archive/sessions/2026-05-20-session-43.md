```

args[]=object%20with%20keys%20%7Bsurface%2C%20reading%2C%20zh_concept%2C%20auto_backfill%7D
```

**Root cause**: `apps/web/src/lib/data/types.ts` declared
`GlossaryEntry.kana_helper: string | null` but the actual runtime payload in
`_fixtures/v1.0.3/glossary.json` is a structured object
`{surface, reading, zh_concept, auto_backfill}` for 308/908 entries (the
other 600/908 are null). Step 7+ Module B layer never observed this drift
because the LLM-facing `assembleScope` only JSON-stringified the field;
Step 11 was the **first TS consumer reading the field** to render in React.
React refused to render an object as a text child → error #31.

Hover.ts SYSTEM_INSTRUCTION line 86 ("If kana_helper.reading is present…")
already encoded the true shape — only the TS type sketch was wrong.

### Fix (3 file edits, same wall, in-source amendments per D-094 §2.1)

1. `apps/web/src/lib/data/types.ts` — introduce `KanaHelper` interface; change `GlossaryEntry.kana_helper` to `KanaHelper | null`.
2. `apps/web/src/lib/glossary/glossaryScope.ts` — `buildGlossarySummary` reads `entry.kana_helper?.reading ?? null` instead of the raw object.
3. `apps/web/src/lib/glossary/__tests__/glossaryScope.test.ts` — first fixture switched from `kana_helper: "プロセッサ"` legacy string to real-shape `{surface, reading, zh_concept, auto_backfill}` object.
