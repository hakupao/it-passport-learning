### Diagnosis (Chrome DevTools MCP `evaluate_script` probe)

```js
// After defensive strip (window.location.href cleaned):
{ baseURI: "https://claude:<pass>@host/quiz?qid=page_042_entity_0",  // ❌ STILL polluted!
  windowLocation: "https://host/quiz?qid=page_042_entity_0",          // ✅ cleaned
  url: "https://claude:<pass>@host/quiz?qid=page_042_entity_0",       // ❌ document.URL polluted
  documentURI: "https://claude:<pass>@host/quiz?qid=page_042_entity_0" }
```

**Root cause**: `history.replaceState` updates `window.location.href` but does **NOT** update `document.baseURI`. Chrome's `Request` constructor resolves relative URLs against `document.baseURI`. Our `quizSseTransport` was using relative `/api/quiz/explain`, so it failed credentials-in-URL even though the visible URL bar was clean.

**Why Step 9 `<Chat />` did not hit this**: useChat (`@ai-sdk/react@3.0.187`) constructs its own absolute URLs internally, bypassing `document.baseURI` resolution.

### Fix: `quizSseTransport.resolveEndpoint`

Constructs absolute URL via `window.location.origin` (which IS strip-respecting per URL spec). Implementation-layer fix; NOT D-NNN-worthy (per D-094 §2.1 + D-080 v1.1 §8 in-source amendment pattern).

`failures/step_10_attempt_1_document_baseuri_credentials_pollution.md` archived (Rule B).
