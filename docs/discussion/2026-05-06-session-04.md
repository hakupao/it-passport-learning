#### Q22 (OQ-04) — 抽检样本量 N 的确定方式

**背景**: 规则 A 说"压缩率 / 改写率 >50% 必须 N 样本独立抽检"。但 **N 是几**？固定？按比例？分层？这是 Topic #5 最核心的元决定，会决定后续每 stage 抽检成本。

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 固定 N** | N = 常数（例如 10），与 stage / 数据规模无关 | 实现最简；评审标准统一；预算可预测 | 大数据集时统计代表性差；小数据集时 N 可能大于 total |
| **(B) 比例 N** | `N = ceil(p × total)`，p 例如 5%，加 floor `min=5` + cap `max=50` | 与数据规模线性挂钩；统计代表性中等 | 需 floor/cap 校正；写入 evidence 时要算 p；超大批次成本超预期 |
| **(C) 分层 N** | 不同 entity type / 不同 stage 各定 N（例如 Term:20 / Question:15 / Table:10） | 与业务实际重要性匹配；难抽到的小类不会"被忽略" | 配置爆炸（per stage × per type）；维护成本高；阈值表需独立 D |

**关键 trade-off**: 实施简单度 ↔ 统计代表性 ↔ 维护成本

**附议**: 这是元决定 —— Q23 (触发 stage 策略) + Q26 (FAIL 处理) 都会用到 N 这个常量 / 公式。

#### Q23 (OQ-26) — 抽检触发的 stage 策略

**背景**: pipeline 8 stage（per D-008）。但不是所有 stage 都"压缩 / 改写 >50%"。需明确**哪些 stage 强制触发抽检**。

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 全 stage 触发** | stage 0~7 每 stage 都抽检 | 最严；零漏检；规则 A 边界外延最大 | 成本翻 8 倍；很多保留性 stage（Unpack / Classify / Export）抽检无意义 |
| **(B) 高改写 stage 触发** | 仅 OCR / Structure / Translate / Audit 强制抽检；其他默认跳过 | 与规则 A 字面"改写率 >50%"对齐；成本可控；语义层级跳变都覆盖 | 需明确"高改写 stage"清单（写入 D）；漏检风险（如某新 stage 加进 pipeline 时容易忘） |
| **(C) 规则 A 字面 / per-stage 自报** | 每 stage 自己测改写率，>50% 才触发；<50% 跳过 | 规则 A 字面执行最忠实；动态适应数据 | 需每 stage 实现"改写率测量"代码；测量本身可能不准；触发不触发不可预期 |

**关键 trade-off**: 严格度 ↔ 成本 ↔ 与规则 A 字面一致度

**附议**: 不管选什么，至少 stage 5 (Translate) 必须强制（三语翻译显然是大幅改写）；这条可以先共识，再选 A/B/C。

#### Q24 (OQ-27) — Reviewer agent 选型 + 模型档位

**背景**: 规则 D 要求 writer / reviewer 不同 `subagent_type` 不同 session（不能自审）。OMC 提供多种 reviewer：`verifier` / `code-reviewer` / `scientist` / `tracer` / `architect` / `critic` 等。**怎么选 / 怎么分配**？

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 单一 reviewer** | 全 stage 用 `verifier`（OMC 默认 reviewer，专责"完成度证据校验"） | 最简；规则 D 自动满足；维护成本零 | 不同 stage 性质各异，单一 reviewer 可能不专业（OCR 改写 ≠ Structure 抽取 ≠ Translate 语义） |
| **(B) 多 reviewer 分工** | 按 stage 性质映射: OCR / Translate → `scientist`（数据/语料）;  Structure → `code-reviewer`（结构/类型）; Audit → `verifier`（完成度收尾）; 跨域因果 → `tracer` | 专业匹配；输出质量更可信 | 心智负担：每加 stage 要决定 reviewer；agent 选错难发现 |
| **(C) stage→reviewer 映射表 + ADR** | 在 `docs/decisions/D-NNN-reviewer-mapping.md` 写一张 stage → reviewer + 模型档位（haiku/sonnet/opus）映射表，作为正式 D | 可追溯；变更走 ADR 流程；与规则 D 形成对照 | 文档负担；映射表需在 stage 增减时同步维护 |

**关键 trade-off**: 简单度 ↔ 专业匹配 ↔ 维护成本

**附议**: **模型档位**也是子问题:

- haiku: 适合规则性、短 context、批量校验（如格式 / 锚点完整性）
- sonnet: 适合中等复杂度、跨段语义（如 Structure / Translate audit）
- opus: 适合深推理、多 hypothesis 比对（如 trace 复杂错误根因）

可以与 reviewer 选型合并决定。

#### Q25 (OQ-28) — evidence 文档字段 + 存放粒度

**背景**: D-030 evidence-template.md 已建（骨架）。要细化两件事:

1. **字段**: 模板里到底填什么字段才能"未来人能复盘"
2. **粒度**: 一份 evidence 文件覆盖多少范围？

**候选 — 字段层级**:

| 选项 | 字段范围 | 优点 | 代价 |
|---|---|---|---|
| **(a) 标准** | stage / sample_id (page+block) / writer_agent / reviewer_agent / 模型版本 / prompt 版本 / PASS-FAIL / 失败原因 / 时间戳 | 可复盘；可比对；reviewer 版本可追溯 | 文档体积中等；要写脚本生成 |
| **(b) 极简** | stage / N / 错误数 / PASS-FAIL | 最少；填表最快 | 失败时无法复盘（不知是哪样本、哪 reviewer、哪 prompt） |
| **(c) 详尽** | (a) + 全文 diff / context 上下文片段 / 抽检 prompt 完整体 | 最全可复盘 | 体积爆炸（一份 evidence 几千行）；检索困难 |

**候选 — 存放粒度**:

| 选项 | 路径形态 | 优点 | 代价 |
|---|---|---|---|
| **(i) per stage 累积式** | `evidence/step_NN_audit.md` 单文件，每 run 追加章节 | 单文件方便检索；与 stage 一一对应 | 文件随 run 数线性膨胀；需文件滚动策略 |
| **(ii) per stage + per run** | `evidence/<run_id>/step_NN_audit.md`，呼应 D-053 累积式 run 目录 | 与 `/data/<cert_id>/runs/<run_id>/` 一一对应；run 隔离干净 | 文件数 × N(stage) × N(run) 爆炸；首次浏览需 cd run 目录 |
| **(iii) per stage + 时间戳滚动** | `evidence/step_NN_audit_<YYYYMMDD-HHMM>.md`，单文件超过 N KB 自动滚 | 文件数可控；老文件可独立归档/压缩 | 时间戳零散；需 index 文件聚合 |

**关键 trade-off**: 可复盘度 ↔ 文档体积 ↔ 检索友好度 ↔ 与 `/data/` 目录约定的对齐度

#### Q26 (OQ-29) — 抽检失败的处理流程

**背景**: PASS 时大家开心。**FAIL 时怎么办**？这一题决定 pipeline 在质量出问题时的容错策略。

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) FAIL 一例 → 全 stage 重跑** | 抽检中任何一个样本 FAIL → 整 stage 重新生成 + 重新抽检 | 最严；零漂流；规则 A 边界最大 | 成本极高；可能死循环（如 reviewer 标准过严）；不区分 sample-level vs stage-level 错因 |
| **(B) FAIL 一例 → 局部修补** | 只针对 FAIL 的样本回炉重新生成，其他样本保留 | 成本最低；快速收敛 | "局部"边界不清（哪个 sample 影响哪些下游）；可能错过系统性 bug |
| **(C) FAIL 比例阈值决策** | PASS 率 ≥X% (例如 90%) → 接受 + 归档失败例 (per 规则 B) + 继续 pipeline；PASS 率 <X% → 全 stage 重跑；任何 FAIL 都必归档 | 平衡严格度和成本；与规则 B 完美对接（FAIL 不删，归档 `failures/`） | 阈值 X% 需独立决定；某些场景不该容忍单例 FAIL（如安全相关字段） |

**关键 trade-off**: 严格度 ↔ 成本 ↔ 数据漂流风险

**附议**: 选 (C) 的话，阈值 X% 是子问题，可与 Q22 (N) 一起算（如 N=10 时 ≥9 PASS = ≥90%）。

### 4.1 Q22 (OQ-04) ✅ — 抽检 N 公式 → B 比例 + bounds（**Claude 决**）

**User 委托原话**:

> "其余的我希望按照你的推荐来，尽量严格，但不能完全把规则定死，我希望可以按照高标准做成整个项目，因为前提是做出来，如果因为标准太高做不出来就尴尬了"

**Claude 推理（user framing 翻译）**:

| User 字面 | Trade-off 偏好 |
|---|---|
| "尽量严格" | bias toward strict |
| "不能完全把规则定死" | 留 escape hatch |
| "前提是做出来" | 拒绝过度严格导致 ship 不出来的方案 |
| "标准太高做不出来就尴尬了" | 拒绝 over-engineering |

按这个偏好对 Q22 三选项重评:

- **A 固定 N**: 与数据规模脱钩（小数据集 N=10=100% / 大数据集 N=10=0.1%），既不严格也不灵活 → **拒**
- **C 分层 N**: 与"严格"对齐但配置爆炸（per stage × per type），首次实施成本太高 → **拒**（可 Phase 2+ 升级）
- **B 比例 N + bounds**: 既严格（与规模线性挂钩）又简单（一公式），加 bounds 防极端 → **选**

**具体公式**:

```
N = max(5, min(30, ceil(0.05 × total)))
```

| 参数 | 值 | 理由 |
|---|---|---|
| 比例 p | 5% | 工业 QC / SQA 业界常见抽检比例（ISO 2859 acceptance sampling 同数量级；HuggingFace evaluate 默认 batch sampling 也类似） |
| floor | 5 | 统计常识：N<5 时 confidence interval 失去意义 |
| cap | 30 | 单 stage 抽检 30 次 LLM 调用 ~ $0.30–$1，单 run 总成本（4 强制 stage × 30 = 120 次）可控 |

**Escape hatch**: pipeline YAML 可 per-stage override，例如:

```yaml
audit:
  default_formula: "max(5, min(30, ceil(0.05 * total)))"
  overrides:
    stage_5_translate:
      N: 50  # translate 涉及术语词典一致性，临时升 N
```

**锁定 D-059 spec**:

- 公式: `N = max(5, min(30, ceil(0.05 × total)))`
- 应用范围: D-060 列出的强制 + 条件抽检 stage
- Override 机制: pipeline YAML `audit.overrides.<stage>.N` 整数覆盖
- 边界: 如 total < 5，N = total（不能多于实际样本）

**Trade-off 备查**:

| 维度 | A 固定 (拒) | B 比例+bounds (选) | C 分层 (拒) |
|---|---|---|---|
| 与数据规模挂钩 | ❌ | ✅ | ✅ |
| 配置复杂度 | 最低 | 低 | 高 |
| 首次实施成本 | 最低 | 低 | 高 |
| 极端兜底 | 无 | floor/cap | 各层独立设 |
| 与 "做出来" 偏好对齐 | 一般 | ✅ | 风险 |

**Rejected reasons**:
- A: 不挂钩规模，违背 "严格" 精神
- C: 配置爆炸，违背 "做出来" 前提（Phase 2+ 可升级）

### 4.2 Q23 (OQ-26) ✅ — 触发抽检 stage → B 高改写 + escape hatch（**Claude 决**）

**Claude 推理**:

按 user "严格但不死板":
- A 全 stage 抽检 = 浪费成本（保留性 stage 抽检无意义）→ **拒**
- C per-stage 自报 = 测量本身不可靠 + 实施成本高 → **拒**
- **B 高改写 stage 显式清单 + escape hatch** = 严格度 + 可控性平衡 → **选**

**强制抽检清单（4 stage + 1 条件）**:

| Stage | 触发性质 | 改写性质 | 必抽理由 |
|---|---|---|---|
| **1 OCR** | 强制 | 像素 → 文本 | OCR 错误是下游所有错误的根源 |
| **3 Hard Re-OCR** | 条件 | 同 stage 1 | 仅 stage 2 标 hard 时跑；跑则强制抽检 |

### 4.5 Q26 (OQ-29) ✅ — FAIL 处理 → C 阈值 + 重跑上限 + 安全字段升级（**Claude 决**）

**Claude 推理**:

按 user "严格但不死板":
- A FAIL 一例 → 全 stage 重跑 = 死循环风险 + 不区分 sample-level vs stage-level 错因 → **拒**
- B FAIL 一例 → 局部修补 = 太松，可能错过 systemic bug → **拒**
- **C FAIL 比例阈值 + 安全字段 escape hatch** = 平衡严格度+成本+安全兜底 → **选**

**PASS 率分档**:

| PASS 率 | verdict | 处理 | 自动度 |
|---|---|---|---|
| ≥90% | **PASS** | ✅ 接受 + 归档失败例 + 继续 | 全自动 |
| 80–90% | **WARN** | ⚠️ 警告 + 归档 + **暂停等 user 决定** | 半自动 |
| <80% | **FAIL** | ❌ 强制全 stage 重跑 + 归档全部 attempt | 全自动 |

**重跑上限**: 同 stage 同 run **max 3 次**；3 次仍不过 → halt + 写 `failures/<cert_id>/runs/<run_id>/RETROSPECTIVE_required.md`，等 user 介入（防死循环 = 守住 user "做出来" 底线）。

**安全字段升级规则（关键 escape hatch）**:

任一安全字段 FAIL → verdict 强制升级为 FAIL（即使 PASS 率 ≥90%）→ 强制重跑（用 opus per D-061）。

**默认安全字段清单**:

| 字段 | 影响 |
|---|---|
| `Question.answer_index` | 题目正确答案 — 错了考试就考砸（产品价值核心） |
| `Term.surface.jp` | 术语日文形 — 错了下游 kana_helper 全错（项目核心动机） |
| `Entity.type` | Discriminated Union discriminator — 错了 D-056 全错 |
| `Envelope.cert_id` | cert 标识 — 错了下游路由全错 (D-058) |
| `Envelope.schema_version` | 版本号 — 错了下游兼容判定全错 (D-058) |

**Escape hatch (YAML)**:

- 阈值可调（e.g. 紧急 ship 时下调到 0.85/0.75）
- 重跑上限可调（1–5）
- 安全字段清单可扩展（per cert 自定义）

**锁定 D-063 spec**:

- 阈值 90% / 80% / <80% 三档
- 重跑 max 3 次（默认）
- 5 项默认安全字段
- 任何 FAIL 必归档（与规则 B 对接）
- 落独立 ADR `docs/decisions/D-063-audit-failure-handling.md`（per D-029）

**Trade-off 备查**:

| 维度 | A 单例触发 (拒) | B 局部修补 (拒) | C 阈值+hatch (选) |
|---|---|---|---|
| 严格度 | 最高 | 最低 | 高 |
| 成本 | 最高 | 最低 | 中 |
| 死循环风险 | 高 | 0 | 0（重跑上限 3）|
| 漂流风险 | 0 | 高 | 低 |
| 与 "做出来" 对齐 | ❌ | ❌ | ✅ |
| 与规则 B 对接 | ✅ | ✅ | ✅ |

**Rejected reasons**:
- A: 单例触发 + 全重跑容易死循环；reviewer 标准过严时永远 ship 不出来
- B: 局部修补错过 systemic bug；如 stage 4 错误是 prompt 缺示例引起，局部补不会发现
