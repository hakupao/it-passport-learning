**Step A.2 design decision (LD-Module-A-1)**: add a **self-report**
binary UI to `<QuizExplain />` footer once `phase === "done"` — "I got
it right" / "I got it wrong" buttons. On click → call helper
`persistQuizOutcome(storage, qid, correct)` (new wrapper around
`recordQuizAnswer + saveProgress`) → buttons show selected state
(idempotent — clicking again is a no-op visually; the underlying
recordQuizAnswer allows overwrite so a user could re-self-report by
re-opening the modal). This is the **honest** answer wire absent a true
picker UX, and additive on D-085 §2.4 frozen contract (modal lifecycle
unchanged — open / explain / close; footer self-report is a leaf addition).

**Step A.1 design decision (LD-Module-A-2)**: `TutorContext` projects the
existing `ChapterSummary[]` (Phase 3 LD-Step1) into 3 mutually-exclusive
status buckets — completed / in-progress / pending. `recentQuiz` is a
sorted-by-lastAnswered-desc slice of `progressStore.quiz` projected to
`{qid, lastAnswered, correct}`. Pure projection — no I/O.

**Step A.3 design decision (LD-Module-A-3)**: `loadTutorContext(storage,
chapters, opts?)` composes `loadProgress(storage)` + the A.1 projection
helpers. SSR-safe via `StorageLike` so the function can be called from a
client component's `useEffect` (after mount-gate) without breaking
hydration — same posture as Phase 3 LD-Step3-D.
