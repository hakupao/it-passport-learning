**Q1 — Phase 2 形态选择 / 路线优先级**

STATE.md §1 的 vision 给了三个候选：
- A = 个人备考工具（CLI / 本地，最小可用）
- B = Web App 题库 / 学习站（公开访问）
- C = AI 学习助手（对话式辅导）
- + 隐含的 D = Phase 5 cert-extractor 通用化（把 Phase 1 框架从 IT Passport 一本书抽出来，支持任意资格教材）

Phase 2 你想 **先做哪一个**？或者 hybrid（比如 A 子集 + C 子集合并成"带 AI 答疑的备考工具"）？还是先做一个完全新方向（比如 Phase 5 通用化前置）？

注意：OQ-05 的措辞是 "A/B/C 三个 Phase 的具体形态 **+ 启动顺序**"，所以本题既要选 Phase 2 是谁，也要顺带说一句 Phase 3/Phase 4 的相对顺序假设（不锁死，只表态）。

**Q2 — RETROSPECTIVE §5.5 16 条 carry-forward 的归属**

retro 列出 16 条 Phase 1 v2 候选（4 类：Stage 6/7 audit 5 / Stage 5 翻译 3 / 架构 4 / 治理 4）。处理方式三选一（或 hybrid）：

- (a) **Phase 1.x patch release 先做**：独立开 Phase 1.5 / 1.6，跑完 16 条（或子集）再发 v1.1.0，然后 Phase 2 起手。优点：cert-extractor 出货质量再升一档；缺点：拖延 Phase 2 entry。
- (b) **Phase 2 第一个 milestone**：Phase 2 起手就先把 16 条吃进来，作为 Phase 2 的 "前置整备" milestone。优点：不开新 phase；缺点：Phase 2 主线被 Phase 1 残务拖。
- (c) **推迟 / 选做子集**：只做关键几条（user 选），剩下封存到 polish_items.json 历史里；Phase 2 直接基于当前 v1.0.2 起步。
- (d) 视 Q1 答案而定：如果 Q1 选了 Phase 5（cert-extractor 通用化），那 16 条几乎全部是通用化前要做的；如果选 A/B/C 应用层，那 16 条很多可以推迟。

**Q3 — 目标用户 & 公开度**

Phase 2 产物的目标用户范围 / 是否公开 / 是否考虑变现，这是设计阶段最影响形态的变量：

- α = 只给你自己备考（私有，不公开；最简单，零隐私 / 法律风险）
- β = 公开但仅 personal-use 性质（开源，不变现；同 Phase 1 模式）
- γ = 公开 + 多用户（要 hosting / 账号 / 数据隔离 / 法律评估；显著放大工作量）
- δ = γ + monetization（订阅 / 付费墙 / GA / 合规等级再升）

这一题决定 Phase 2 是否需要 GUI / 服务端 / 多用户 / 法律评估。也决定 cert-extractor 本身的版权范围怎么处理（当前 v1.0.2 已脱敏书名 / 作者，但内容仍源自一本受版权保护的教材；β/γ/δ 都会让这个问题更尖锐）。
