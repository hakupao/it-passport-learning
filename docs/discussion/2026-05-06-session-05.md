#### Q27 (OQ-30) — entry_points 命名空间设计

**背景**: Python entry_points 通过 group name 注册（如 `console_scripts` / `pytest11`）。cert-extractor 有 4 个 axis（Source / OCR / Translator / Exporter），每个轴可注册多个插件。**group name 怎么设计**？

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) 单 namespace** | 所有插件统一注册到 `cert_extractor.plugins`；axis 通过插件 metadata 区分 | 配置最简；与单一插件入口对齐 | 加载时要遍历所有插件读 metadata 才能分类；类型不严 |
| **(B) 4 个 axis-namespace** | `cert_extractor.source` / `.ocr` / `.translator` / `.exporter` 各自独立 | 类型严格；按 axis 加载时直接拿 group；与 D-021 4 axis 模型 1:1 | 加 axis（如未来加 `validator`/`enricher`）需扩展 group 列表 |
| **(C) 嵌套式** | `cert_extractor.plugins.source` / `.plugins.ocr` 等（带 `plugins.` 中间路径） | 命名空间清晰（`plugins.*` 一望而知是插件） | 字符更长；与 (B) 实质等价；业界惯例较少 |

**关键 trade-off**: 类型严格度 ↔ 加 axis 时的迁移成本 ↔ 与业界 entry_points 惯例对齐

**业界参考**:
- pytest 用 `pytest11`（单 namespace）
- Flask 扩展用 `flask.commands` 等（按用途分 namespace = B 模式）
- Django apps 不走 entry_points（自己的注册机制）

---

#### Q28 (OQ-31) — Phase 1 阶段是否启用 entry_points

**背景**: D-026 字面"第三方走 entry_points"，但 **Phase 1 没有第三方，只有内部插件**。问题: Phase 1 阶段 entry_points 启不启用？

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) Phase 1 双轨** | 内部插件用 D-025 装饰器（`plugins/` 目录扫描），外部插件用 entry_points；两套并行 | 内部不付 entry_points 开销；和 D-026 字面对齐 | 双套加载逻辑；测试 entry_points 路径要等 Phase 5 真有第三方时 |
| **(B) Phase 1 only D-025** | entry_points 完全不启用，留到 Phase 5；现在只走装饰器 + 目录扫描 | 实施最简；Phase 1 ship 最快 | entry_points 路径未经实战测试，Phase 5 启用时可能有坑；D-026 部分推迟 |
| **(C) 全统一走 entry_points** | 内部插件也通过 entry_points 注册（在 `packages/extractor/pyproject.toml` 里写自己的 entry_points）；D-025 装饰器只是注册的语法糖 | 单一加载机制；Phase 1 就把 entry_points 路径走顺；第三方加入零摩擦 | 内部插件发布时也要 reinstall（`uv sync` / `pip install -e`）才能让 entry_points 生效；开发体验略差 |

**关键 trade-off**: Phase 1 实施成本 ↔ 第三方友好度 ↔ 一致性

**附议**: 这一题是 Topic #6 的"meta 元决定"，影响 Q27 / Q29 / Q30 是否都要在 Phase 1 实化。

#### Q29 (OQ-32) — 版本协商失败行为

**背景**: D-026 锁了"插件 `__cert_extractor_min_version__` 单行声明"，但**协商失败时怎么办**没定。

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) Fail-fast** | 加载时若 `lib_version < plugin.min_version` → raise + halt pipeline | 最严；错误第一时间暴露；与 D-057 全 stage 强校验风格一致 | 一个废插件让整 pipeline 启动不了；user 体验差 |
| **(B) Warn + 跳过** | 不兼容插件 → log warning + 跳过加载；其他插件继续 | 用户体验好；部分功能仍可用 | 静默漂流风险（user 可能没注意 warning，pipeline 跑出错产物） |
| **(C) `strict_mode` flag** | pipeline YAML 配置 `plugins.strict_mode: true/false`；true=fail-fast，false=warn+skip；默认 true | 严格度可调；符合 user "严格但不死板" 偏好 | 配置增加；user 可能误关 strict_mode 后忘了 |

**子题**:

1. 是否支持 `max_version`？（plugin 能声明"我只兼容到库 1.x"防止跨 MAJOR）
2. 是否支持 version range？（`>=0.5,<2.0` 类似 PyPI dependency）

**关键 trade-off**: 严格度 ↔ 用户体验 ↔ 配置复杂度

#### Q30 (OQ-33) — name 冲突解决机制

**背景**: 两个第三方插件都注册了 OCR axis 同名 `tesseract` 怎么办？或者第三方插件名字撞上内部插件 `mistral`？

**候选**:

| 选项 | 形态 | 优点 | 代价 |
|---|---|---|---|
| **(A) First-wins** | 按 entry_points 加载顺序，第一个赢 | 实现最简 | 加载顺序非确定（依包名 alphabetical），用户难预测 |
| **(B) Last-wins** | 后加载者覆盖前者 | 第三方插件可主动 override 内部 | 内部插件被静默替换风险；调试难 |
| **(C) Raise error** | 任一冲突 → raise；强制 user 在 pipeline YAML 显式 disambiguate | 安全；user 必须 aware | 用户体验差（一冲突就 halt）；首次设置成本高 |
| **(D) Package-prefix 强制** | 注册 name 必须带 package 前缀（如 `paddleocr::tesseract`）；冲突自然消失 | 完全无冲突；displayed name 可短 | 命名规则强制；对 Phase 1 内部插件也要带前缀? |

**关键 trade-off**: 简单 ↔ 显式 ↔ 防止意外 override ↔ 用户认知负担

#### Q31 (OQ-34) — 插件元数据 + config_schema 范围

**背景**: 除 D-026 锁的 `__cert_extractor_min_version__`，插件还要声明哪些 metadata？是否要带 config_schema（Pydantic 模型描述插件接受什么 config）？

**候选**:

| 选项 | 字段范围 | 优点 | 代价 |
|---|---|---|---|
| **(A) 极简** | name / description / `__cert_extractor_min_version__`（单行声明） | 第三方接入门槛最低；实施最快 | 自动文档生成 / IDE 友好度差；插件配置错误难定位 |
| **(B) 标准** | (A) + `supported_axes: list[str]` + `config_schema: type[BaseModel]` + `author` + `license` + `homepage` | 自动文档生成；config 类型校验；License/author 法律合规 | 插件作者要写更多代码；维护成本中等 |
| **(C) 详尽** | (B) + `documentation_url` + `examples: list[dict]` + `test_fixtures: list[Path]` + `changelog: str` | 第三方插件市场友好；可生成 plugin index / 网站 | 第三方接入门槛高；过度规约可能阻止小插件 |

**关键 trade-off**: 维护成本 ↔ 第三方友好度 ↔ Phase 1 实施成本 ↔ 自动文档 / 类型校验 / IDE 体验

**附议**: 如果选 (B)/(C)，`config_schema` 字段使插件配置可以走 Pydantic v2 校验（呼应 D-057 全 stage 强校验），与项目整体风格一致。

### 4.1 Q27 (OQ-30) ✅ — entry_points 命名空间 → B 4-axis-namespace（**Claude 决**）

**Claude 推理**:

按 user framing "严格但不死板":
- A 单 namespace `cert_extractor.plugins` = 加载时遍历所有 plugin 读 metadata 才能分类 → 类型不严，违 "严格"
- C 嵌套式 `cert_extractor.plugins.source` = 与 B 实质等价但字符更长，业界惯例少
- **B 4 axis-namespace** = 与 D-021 4 axis 模型 1:1 + 加载性能好（按 axis 直接拿 group） + 类型严格

**业界参考**:

| 项目 | namespace 模式 | 备注 |
|---|---|---|
| Flask | `flask.commands` / `flask.blueprints` | 按用途分 namespace = B 模式 |
| SQLAlchemy | `sqlalchemy.dialects` / `.plugins` | 按 axis 分 |
| pytest | 单 `pytest11` | A 模式（单一插件接口） |
| Trio | `trio.abc.AsyncResource` 等 | 按 abstract type 分 |

→ B 与 Flask / SQLAlchemy 主流对齐；pytest 单 namespace 是因其插件类型单一，不适合 cert-extractor 4 axis 场景。

**锁定 D-064 spec**:

| Axis | entry_points group |
|---|---|
| Source | `cert_extractor.source` |
| OCR | `cert_extractor.ocr` |
| Translator | `cert_extractor.translator` |
| Exporter | `cert_extractor.exporter` |

加新 axis（如未来 `cert_extractor.validator` / `.enricher`）只需新加 group，不破坏现有。

**Trade-off 备查**:

| 维度 | A 单 ns (拒) | B 4-axis (选) | C 嵌套 (拒) |
|---|---|---|---|
| 类型严格度 | ❌ | ✅ | ✅ |
| 加载性能 | 差（要遍历过滤） | 优（直接按 group 拿） | 优 |
| 加 axis 迁移成本 | 0 | 低（加 1 group） | 低 |
| 业界对齐度 | pytest | Flask / SQLAlchemy | 较少 |
| group name 长度 | 短 | 中 | 长 |

**Rejected reasons**:
- A: 加载时要遍历 + 过滤 metadata，违 "严格" + 性能差
- C: 与 B 实质等价但 verbose，无收益

### 4.2 Q28 (OQ-31) ✅ — Phase 1 是否启 ep → B + 接口预留 ★ 元决定（**Claude 决**）

**Claude 推理**:

按 user "前提是做出来":
- A Phase 1 双轨 = 实施 entry_points 路径但无调用方测试 → 拒（Phase 5 启用时仍可能有坑）
- C 全统一走 entry_points = 内部 plugin 改完要 reinstall，开发体验差 → 拒
- **B Phase 1 only D-025** = 最简

但 Claude 自决加细化: **plugin loader 内部预留 entry_points 接口**（feature flag / 注释开关），Phase 1 不启用，Phase 5 打开 flag 即可。这样:
- Phase 1 不付双轨成本
- Phase 5 transition 从 "重构" 降为 "打开 flag"
- D-026 字面对齐（"内部用装饰器，第三方走 entry_points"，本 D 锁了实施时机）

**实施期参照**:

```python
# packages/extractor/src/cert_extractor/plugins/loader.py
class PluginLoader:
    def __init__(self, enable_entry_points: bool = False):
        self.enable_entry_points = enable_entry_points

    def load_all(self) -> dict:
        plugins = {axis: {} for axis in AXES}
        plugins = self._merge(plugins, self._load_via_decorators())
        if self.enable_entry_points:
            plugins = self._merge(plugins, self._load_via_entry_points())
        return plugins
```

**锁定 D-065 spec**:

- Phase 1: plugin loader 只走 D-025 装饰器路径（`_load_via_decorators` 唯一调用）
- entry_points loader 路径 (`_load_via_entry_points`) 在代码里预留，Phase 1 默认 `enable_entry_points=False`
- Phase 5 启用时只需将 flag 改为 True（pipeline YAML 控制或硬编码切换）
- 内部插件**不**在 `pyproject.toml [project.entry-points]` 注册（保持 D-025 装饰器为唯一注册路径）
- 此决定独立 ADR `docs/decisions/D-065-plugin-loading-strategy.md`（meta 决定，跨 Phase 影响）

**Trade-off 备查**:

| 维度 | A 双轨 (拒) | B + 接口预留 (选) | C 全统一 (拒) |
|---|---|---|---|
| Phase 1 实施成本 | 翻倍 | 最低 | 中 |
| 内部开发体验 | 好 | 好 | 差（reinstall） |
| Phase 5 transition 成本 | 0（已实施） | 最低（打开 flag）| 0 |
| 实战测试覆盖 | 双轨 0 调用方 | Phase 1 决定路径有覆盖 | 单一路径有覆盖 |
| 与 D-026 字面对齐 | 全面但太早 | 完美（按 Phase 推进） | 偏离（内部也走 ep） |

**Rejected reasons**:
- A: entry_points 路径 Phase 1 无调用方，未经实战测试；Phase 5 启用时仍可能有坑
- C: 内部 plugin 改完要 reinstall (`uv sync`)，违反 user "做出来" (开发速度被拖累)

### 4.3 Q29 (OQ-32) ✅ — 版本协商失败 → C + 默认 strict（**Claude 决**）

**Claude 推理**:

按 user "严格但不死板":
- A 默认 fail-fast = 严格 ✓
- B 默认 warn+skip = 不死板 ✓
- → **C strict_mode flag** = 两者兼得

默认 `strict_mode = true`（fail-fast 默认），与 D-057 全 stage 强校验风格一致；user 可在 pipeline YAML 关掉变成 warn+skip。

**子题处理**:

| 子问题 | 决策 | 理由 |
|---|---|---|
| 支持 `max_version`? | **是 (Phase 5 实化)** | 跨 MAJOR 风险大；Phase 1 内部插件用不到 |
| 支持 version range? | **否 (YAGNI)** | min/max 已够用；range 增加解析复杂度无收益 |

**锁定 D-066 spec**:

- pipeline YAML key: `plugins.strict_mode: bool` (默认 `true`)
- `strict_mode = true`: 任一 plugin `lib_version < plugin.min_version` → raise + halt pipeline
- `strict_mode = false`: 不兼容 plugin → log warning + 写 `failures/` + skip 该 plugin（其他 plugin 继续）
- `__cert_extractor_min_version__: str` — 必须 (D-026 已锁)
- `__cert_extractor_max_version__: str | None = None` — 可选 (Phase 5 实化)
- 不支持 version range / wildcards / `~=` 等

**Trade-off 备查**:

| 维度 | A fail-fast (拒) | B warn+skip (拒) | C strict_mode (选) |
|---|---|---|---|
| 默认严格度 | 最高 | 最低 | 高（默认 strict）|
| 用户体验 | 差（一插件错全 halt） | 好（漂流风险）| 中（默认严但可关）|
| 与 D-057 全 stage 强校验风格 | ✅ | ❌ | ✅（默认）|
| 与 user "不死板" 对齐 | ❌ | ✅ | ✅ |
| 配置复杂度 | 0 | 0 | 1 个 flag |

**Rejected reasons**:
- A: 一个废插件让整 pipeline 启动不了，违 user "做出来"
- B: 默认 warn+skip = 静默漂流风险，违 "严格"

### 4.4 Q30 (OQ-33) ✅ — name 冲突 → C + YAML disambiguate（**Claude 决**）

**Claude 推理**:

按 user "严格但不死板":
- A First-wins = 加载顺序 alphabetical 不可预测，违 "严格"
- B Last-wins = 内部插件被静默 override（危险），违 "兜底" 精神
- D Package-prefix 强制 = verbose（每次 name 都得带前缀），违 "做出来"
- **C Raise + YAML disambiguate** = 默认严格 + YAML 留 hatch

具体:

```yaml
# 默认场景（无冲突）
plugins:
  ocr:
    use: "tesseract"   # 唯一注册 → 直接用

# 冲突场景（多 package 都注册了 "tesseract"）
plugins:
  ocr:
    use: "paddleocr_pkg::tesseract"   # 显式 disambiguate package
```

**锁定 D-067 spec**:

- 默认: 同 axis 同 name 由多 package 注册 → raise + halt + 提示 user 在 YAML disambiguate
- pipeline YAML 可写 `<package>::<name>` 强制显式选
- 内部插件 implicit package = `cert_extractor`，等价于 `cert_extractor::<name>`
- 错误消息必须列出所有冲突 package 路径，让 user 知道选哪个

**Trade-off 备查**:

| 维度 | A first-wins (拒) | B last-wins (拒) | C+YAML (选) | D prefix 强制 (拒) |
|---|---|---|---|---|
| 严格度 | 低 | 低 | 高 | 最高 |
| User UX | 差（不可预测）| 危险（静默 override）| 中（首次冲突需配 YAML）| 差（每次都 verbose）|
| 防意外 override | ❌ | ❌ | ✅ | ✅ |
| 留 hatch | 无 | 无 | YAML | 无 |

**Rejected reasons**:
- A: 加载顺序 alphabetical → user 难预测
- B: 内部插件可能被静默 override，难调试
- D: name 总要带 prefix 太 verbose，开发体验差

### 4.5 Q31 (OQ-34) ✅ — 元数据 + config_schema → B 标准（**Claude 决**）

**Claude 推理**:

按 user "严格但不死板，前提是做出来":
- A 极简 = 没 config_schema → 配置错误难定位，违 "严格"
- C 详尽 = 接入门槛高，违 "做出来"
- **B 标准** = sweet spot

`config_schema` 是关键: 插件作者写一个 Pydantic BaseModel 描述插件接受什么 config，与 D-057 全 stage 强校验风格一致。Plugin loader 自动用此 schema 校验 pipeline YAML 中 plugin section，非法 config → raise。

**锁定 D-068 spec — 标准元数据集**:

```python
# 插件 contract（实施期参照）
from pydantic import BaseModel, ConfigDict

class TesseractConfig(BaseModel):
    """plugin 自定义 config schema."""
    model_config = ConfigDict(extra="forbid", strict=True)
    languages: list[str] = ["jpn"]
    psm: int = 3


class TesseractPlugin:
    # ===== 必填字段 =====
    name: str = "tesseract"
    description: str = "Tesseract OCR engine for Japanese kana / kanji"
    supported_axes: list[str] = ["ocr"]  # 一个 plugin 可能跨 axis
    config_schema: type[BaseModel] = TesseractConfig

    # D-026 单行声明
    __cert_extractor_min_version__: str = "0.5.0"

    # ===== 推荐字段（可选但强烈建议）=====
    author: str = "..."
    license: str = "Apache-2.0"
    homepage: str = "https://..."

    # ===== 可选 (Phase 5 实化) =====
    __cert_extractor_max_version__: str | None = None
```

加载时 plugin loader:
1. 读 `__cert_extractor_min_version__` (per D-066) → 校验
2. 读 `name` / `supported_axes` (per D-064 / D-067) → 注册到 axis-namespace
3. 读 `config_schema` → 校验 pipeline YAML 中对应 section（per D-057 强校验）
4. `author / license / homepage` → 用于自动生成 plugin index 文档

**Trade-off 备查**:

| 维度 | A 极简 (拒) | B 标准 (选) | C 详尽 (拒) |
|---|---|---|---|
| 第三方接入门槛 | 最低 | 中 | 高 |
| 配置错误定位 | 差（无 schema） | ✅（Pydantic 校验） | ✅ |
| 自动文档 / IDE | 差 | ✅ | ✅ |
| 法律合规 (license / author) | ❌ | ✅ | ✅ |
| 与 D-057 强校验风格 | ❌ | ✅ | ✅ |
| 与 user "做出来" 对齐 | ✅ | ✅ | ❌ |

**Rejected reasons**:
- A 极简: 没 config_schema → 配置错误要等运行时才暴露 + 违 D-057
- C 详尽: examples / test_fixtures / changelog 等增加接入成本，可在 Phase 5 真有 plugin 市场需求时再加
