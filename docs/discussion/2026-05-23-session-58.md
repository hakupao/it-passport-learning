### Q1 — Surface form (C.1 core decision)

What is the tutor surface form?

- (a) **Standalone `/[locale]/tutor` route** — 5th NavTabs slot, full-page
      chat UI, deepest immersion + clearest mental model ("a place to
      study with the tutor"). Cost: +1 route + NavTabs C.4 work + chrome
      space pressure (already 4 tabs).
- (b) **Modal inside book trunk** — opens from chapter-end-panel 3rd
      action button (alongside 问本章 / 测本章). TutorContext can default
      to current chapter scope; no new route; chrome stays at 4 tabs.
      Cost: tutor is "in-book only" — harder to reach without entering
      a chapter first; competes with `<ChapterChatModal />` for the same
      panel slot.
- (c) **Hybrid** — standalone primary + book-context "open tutor here"
      affordance that pre-fills chapter scope. Cost: more code +
      duplicated entry surface; benefit = both immersion modes.
- (d) **You decide / 你来定** — Claude consults D-101 (book trunk
      decision) + D-102 §7 (form discussion) + Phase 3 LD-1 (NavTabs
      hierarchy) + Context7 if useful, locks D-NNN with rationale +
      rejected alternatives.

### Q2 — Conversation persistence model (C.2 contract)

How should tutor conversation history persist?

- (a) **Single global thread** — one continuous learning conversation
      across visits. Resume-style UX (mirrors Phase 2 chat `historyStore`
      precedent but with separate key `itp:tutor:session:v1` so D-085
      §2.4 stays untouched). Pro: continuity. Con: thread can grow
      unbounded → context window pressure.
- (b) **Per-chapter scoped thread** — each chapter has its own
      conversation, addressed by `nn`. Pro: cleaner pedagogical context;
      no bleed across chapters. Con: 16 separate threads → state to
      manage + UX to switch.
- (c) **Ephemeral** — forget on close/refresh. Pro: privacy-first;
      lightest implementation; no schema-evolution risk. Con: no
      continuity across visits.
- (d) **User-toggleable** — "save conversation" checkbox; default off.
      Pro: respects user choice. Con: more UI; defers the decision.

### Q3 — TutorContext refresh policy (cache stability fork)

When should `loadTutorContext()` re-fire during a tutor session?

- (a) **Load on mount only — snapshot** — TutorContext frozen at session
      open. **Best for Anthropic ephemeral cache stability** (≥80%
      target per D-103 §2.4 + B.3 verified) — the inner preamble
      breakpoint stays byte-stable across turns within a session. Con:
      progress made mid-session not reflected until reload.
- (b) **Refresh every send** — always current progress; cache key drifts
      more (every turn = new inner preamble bytes → inner cache breakpoint
      may invalidate; outer SYSTEM breakpoint still hits per LD-Module-B-13
      bulk). Pro: tutor always knows your latest state. Con: cache hit
      ratio degrades; cost rises proportionally.
- (c) **Explicit "refresh context" button** — user controls; default
      = snapshot. Pro: both worlds. Con: more UI; most users won't
      press it.
- (d) **You decide / 你来定**.

### Q4 — Escalation UX (D-104 §2.1 `escalate` flag wire)

How does the user trigger escalation (DeepSeek max thinking effort /
Anthropic Opus 4.7)?

- (a) **Auto-detect** — LD-Module-B-1 heuristic (user-explicit
      harder-reasoning ask OR retry-after-low-confidence). No UI
      control; tutor decides. Pro: cleanest UX. Con: heuristic is
      brittle; "user-explicit" needs NLP-detection or keyword match
      (over-engineered for v1).
- (b) **Explicit user toggle** — "deeper reasoning" button next to
      send; sticky-per-message or sticky-per-session. Pro: clear
      control + clear cost model (user opts into the higher tier).
      Con: 95% of users won't touch it.
- (c) **Both** — auto suggest + user override.
- (d) **Hide escalate UI for v1** — only ever send `escalate=false`;
      escalate becomes a backend-only knob reserved for Phase 5
      adaptive recommendation work (D-102 §7.2 deferred). Phase 4 ships
      with Sonnet 4.6 / V4 pro defaults only. Pro: simplest v1;
      preserves D-104 envelope intact; matches "α-private no-stakes
      tutor" posture. Con: escalate codepath untested end-to-end until
      Phase 5.
