## 失败模式分类 / Failure Mode

- [x] `model-bug` — confirmed not chunk-size-driven (halving chunk
  from 8 → 4 only reduced fail count by ~28%, far less than the 50%
  output-tokens reduction would predict). Some other property of these
  particular pages is causing the model to refuse / truncate.
- [x] `prompt-issue` — likely. Hypothesis: the inlined glossary in the
  system prompt + the request structure may cause opus to "deduplicate"
  outputs whose jp surface is similar to a glossary key (e.g. page_031's
  ent[0] title `株式会社と経営理念` literally contains the glossary key
  `経営理念` as a substring). The model may be returning *only* the
  glossary translation for that surface and skipping the wrapper, then
  the parser drops the malformed item.
- [ ] (other categories unchanged from 001)

**Pattern observation**: the 4 still-failing items on each page are
**consecutive in entity index** AND mix short + long. This suggests
they share a prompt-level property — most likely they were dispatched
as a single sub-batch (4 items at chunk=4) where the model returned
0 / 4 items.
