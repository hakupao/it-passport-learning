## 技术判定 / Technical Verdict

**FAIL** — 11 of 393 Trilingual leaves still bear the `<UNTRANSLATED>`
sentinel. Stage 7 export per spec refuses any envelope with the
sentinel, so this run is not pipeline-acceptable.

The runner itself behaved correctly: it surfaced the failure as
`fail_count=11` in `Stage5Result`, with 11 entries of the form
`page_NNN: skipped jp=… reason=model returned short batch` in
`failures`. The error is purely model-side.

---

## 业务判定 / Business Verdict

**FAIL** — Pages 031 + 038 are content pages whose definitions are
exactly the kind of long-form learning material the trilingual factory
exists to produce. A learner reading the rendered output would see the
locked term surface but no Chinese / English explanation, defeating
the core value proposition.

The 11 missing leaves are all **`definition` / `caption` / `title`**
fields — the long-prose ones the model truncated. None of the failing
leaves are short labels or glossary-substring matches.

---

## 失败模式分类 / Failure Mode

- [x] `model-bug` — opus stably returns a short JSON array when asked
  to translate ≥6 long-form definitions in one batch with the inlined
  glossary system prompt. Other 31 content pages with shorter items
  per batch passed cleanly.
- [x] `prompt-issue` — current `USER_PROMPT_TEMPLATE` does not
  explicitly instruct "return exactly N items, all of them" as a
  precondition. Adding a stricter contract may help.
- [ ] `threshold-issue`
- [ ] `data-issue`
- [ ] `infrastructure`
- [ ] `glossary-mismatch` — glossary hits within these pages were all
  resolved correctly; the failures are exclusively non-glossary leaves.
- [ ] `schema-violation` — produced JSON parsed cleanly; it was just shorter than requested.
- [ ] `regression`
- [ ] `other`

The 11 / 393 = 2.8% leaf-level failure rate concentrates entirely on 2
of 32 content pages (6.3% page-level failure rate). The other 30
content pages (e.g. page_045 with 38 leaves over 5 sub-batches)
completed without short-batch issues, suggesting this is sensitive to
**average jp item length within a sub-batch** rather than batch count.
